## System Architecture

The ecosystem consists of a multi-layer, distributed infrastructure that ensures the collection, processing, persistence, and visualization of data generated by a hardware simulation (Marsami agents and environmental sensors):

1. **IoT & Simulation Layer (`mazerun.exe`):** Asynchronous emission and reception of events (movements, temperature/noise alerts, and actuator commands) via the **MQTT** protocol.
2. **Ingestion & Automation Layer (Python):** - Dedicated scripts for MQTT topic subscription and immediate persistence.
   - Terminal orchestrator (`iniciar_terminais.py`) and background daemons (`vigilante_windows.py`) for state control and *always-on* automation.
3. **Hybrid Persistence Layer (Polyglot Persistence):**
   - **MongoDB (NoSQL):** High-availability repository for raw telemetry data and analytical auditing.
   - **MySQL (Relational):** Transactional database for corporate business logic, access control, and state synchronization.
4. **Monitoring & Control Layer (Android & PHP):**
   - **Web Portal (PHP):** Configuration, parameterization, and operational control of the simulation.
   - **Mobile App (Native Android - Java):** Analytical KPI dashboard with real-time charts and control room (NOC) monitoring.

## Technical Highlights & Engineering Decisions

### 1. Optimization via Data Shedding (Graceful Degradation)
High-frequency IoT systems generate massive volumes of redundant micro-variations. To protect disk I/O and prevent database locking, we implemented a **Data Shedding** strategy. The system deliberately discards continuous and identical temperature and sound readings ("white noise"), prioritizing bandwidth and the processing of critical alerts and state transitions. We traded irrelevant micro-precision for absolute systemic stability.

### 2. Autonomous Migration Pipeline & Self-Healing
We avoided the common pitfall of centralizing the system's state in the volatile RAM of Python scripts (preventing a *Single Point of Failure*). All state logic was delegated to the databases. In the event of a crash or forced machine reboot, the Python script connects to MongoDB and **automatically reconstructs the history and moving averages** from the database aggregations, resuming operations exactly where it left off without any data loss.

### 3. Operational Hysteresis Mechanism (Actuator Control)
To mitigate erratic sensor oscillation at critical threshold boundaries (*chattering*), the system rejects static linear control. Through the `NivelReabertura` parameter, we introduced a **Hysteresis** window. If the noise threshold is 70dB, the doors lock immediately at the peak of danger, but the system creates a safety "deadband": the doors only reopen when the environment stabilizes and proves safe (e.g., below 50% of the threshold = 35dB). This protects physical actuators and pacifies the MQTT network.

### 4. Backend Alert Anti-Spam Shield
The `SP_RegistarAlerta` procedure features a native event suppression layer based on a user-parameterized dynamic time interval (*cooldown*). If a sensor fails and starts bombarding the network with hundreds of identical or consecutive readings, the MySQL engine acts as a protective shield, suppressing the spam and recording only data that represents a real evolution of the system's state.

### 5. Advanced Security (RBAC & IDOR Mitigation)
Security was implemented at the DBMS kernel level, utilizing a strict Role-Based Access Control (**RBAC**) model. 
- The `utilizador` table **does not store passwords**; authentication is natively delegated to the MySQL driver via dynamically created physical accounts (`CREATE USER`).
- Stored Procedures (such as `SP_EditarPerfil`) cross-reference the secure session token ID with the ID contained in the JSON payloads. If there is a mismatch, the transaction is aborted due to privilege failure, natively mitigating **IDOR (Insecure Direct Object Reference)** attacks.

## Stored Procedures Structure (MySQL)

The transactional core of the system is encapsulated in 17 highly optimized Stored Procedures:

- **`SP_RegistarPassagem`:** Marsami movement and fatigue tracking, controlling occupancy by parity (Odd/Even Rooms).
- **`SP_RegistarAlerta`:** Hardware alert ingestion with integrated anti-spam logic.
- **`SP_TerminarSimulacao`:** Official termination of the active session and end timestamp recording.
- **`SP_EditarPerfil`:** Dynamic data update via JSON and synchronization with `RENAME USER` in the MySQL engine.
- **`SP_ApagarUtilizador`:** Complete deletion of the business account and secure destruction of native credentials (`DROP USER`), preventing orphan accounts.
- **`SP_CriarUtilizador`:** Account registration with regex validation (phone/team) and dynamic application of the privilege matrix (differentiated `GRANT`s for Admin, User, Monitor, and Migrator).
- **`SP_ValidarLogin`:** App hydration with metadata following a secure native connection.
- **`SP_CriarSimulacao`:** Initialization of a simulation in the configuration state (State 0).
- **`SP_ObterHistorico`:** Global analytical consolidation uniting operational and historical tables via `UNION ALL`.
- **`SP_ValidarParametros`:** Geometric and logical threshold validation prior to starting the simulation.
- **`SP_IniciarSimulacao`:** Transition to State 1, clearing of live operational tables, and reset of `AUTO_INCREMENT` counters.
- **`SP_EstatisticasFinaisSimulacao` & `SP_GetResumoFinal`:** Real-time calculation and consolidation of KPIs (MVP Marsami, Top Explorer, Traffic Hotspot Rooms, and Duration).

##  How to Run

### 1. Prerequisites
- Active MySQL server and MongoDB instance.
- Accessible MQTT Broker (local or cloud, e.g., `broker.hivemq.com`).
- Simulation executables (`mazerun.exe` and `monitmaze.exe`).

### 2. Database Setup
1. Import the structural schema found in `labirinto_DB-18.sql` into your MySQL server.
2. Ensure the `root` user has permissions to execute user manipulation commands (`CREATE USER`, `GRANT`, `RENAME USER`).

### 3. Running the Scripts
1. Configure the access credentials and broker addresses in the Python configuration files.
2. Activate the local monitoring daemon:
   ```bash
   python vigilante_windows.py
